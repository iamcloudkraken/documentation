# Amazon S3 Sink

The S3 sink writes data to Amazon S3 (or any S3-compatible storage) as partitioned files. It supports time-based directory hierarchies, configurable file rolling policies, and multiple output formats with optional compression.

## Required Properties

| Property | Type | Required | Description |
|----------|------|----------|-------------|
| `bucket` | string | Yes | Target S3 bucket name |
| `region` | string | Yes | AWS region (e.g., `us-east-1`) |
| `connectionRef` | string | Yes | Reference to an AWS `Connection` resource |
| `prefix` | string | No | Static path prefix before the partition pattern |
| `partitionPattern` | string | No | Time-pattern for directory partitioning (default: `yyyy/MM/dd/HH`) |
| `format` | string | Yes | Output format: `parquet`, `json`, `csv`, `avro` |
| `compression` | string | No | Compression codec (see table below) |
| `rolling` | object | No | File rolling policy (size / interval based) |

## Partitioning

STREAMINGPLUS uses event-time partitioning by default, deriving the partition path from the record's `eventTime` field. You can override this with a custom field via `partitionKey`.

```yaml
spec:
  sink:
    type: s3
    connectionRef: aws-prod
    bucket: my-data-lake
    region: us-east-1
    prefix: events/
    partitioning:
      pattern: "yyyy/MM/dd/HH"         # → events/2024/06/15/14/
      fieldOverride: eventTimestamp     # use this record field instead of eventTime
      timezone: UTC
```

This produces paths such as:

```
s3://my-data-lake/events/2024/06/15/14/part-0001.parquet
```

### File Rolling Policy

Files are rolled (closed and a new file opened) based on either elapsed time or accumulated size, whichever is reached first:

```yaml
rolling:
  intervalSeconds: 300    # roll every 5 minutes
  maxFileSizeBytes: 134217728  # or when file reaches 128 MB
  inactivityTimeoutSeconds: 60 # roll if no new records for 60 s
```

:::tip
For downstream Athena or Glue crawlers, keep `intervalSeconds` at 300–900 and `maxFileSizeBytes` at 128–512 MB to produce files that are efficiently queried.
:::

## Compression Options

| Format | Supported Codecs | Notes |
|--------|-----------------|-------|
| `parquet` | `snappy` (default), `gzip`, `zstd`, `none` | Snappy recommended for query performance |
| `json` | `gzip`, `zstd`, `none` | Line-delimited JSON (NDJSON) |
| `csv` | `gzip`, `none` | Header row written per file |
| `avro` | `snappy`, `deflate`, `none` | Schema embedded in each file |

## Full YAML Example

```yaml
apiVersion: streamingplus.io/v1
kind: Deployment
metadata:
  name: events-to-s3
  namespace: production
spec:
  source:
    type: kafka
    connectionRef: kafka-prod
    topics:
      - user-events
  transforms:
    - type: filter
      expression: "record.type != 'heartbeat'"
    - type: flatten
      fields:
        - metadata
  sink:
    type: s3
    connectionRef: aws-prod
    bucket: my-data-lake
    region: us-east-1
    prefix: events/
    partitioning:
      pattern: "yyyy/MM/dd/HH"
      timezone: UTC
    format: parquet
    compression: snappy
    rolling:
      intervalSeconds: 300
      maxFileSizeBytes: 134217728
      inactivityTimeoutSeconds: 60
    errorHandling:
      deadLetterSink:
        type: kafka
        connectionRef: kafka-prod
        topic: streamingplus.dead-letter.s3
      maxRetries: 5
      retryBackoffMs: 2000
    bufferConfig:
      flushIntervalMs: 10000
      maxBufferedRecords: 50000
```

## IAM Permissions

The AWS identity used by the `Connection` resource must have the following minimum S3 permissions:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:PutObject",
        "s3:AbortMultipartUpload",
        "s3:ListMultipartUploadParts"
      ],
      "Resource": "arn:aws:s3:::my-data-lake/*"
    },
    {
      "Effect": "Allow",
      "Action": "s3:ListBucket",
      "Resource": "arn:aws:s3:::my-data-lake"
    }
  ]
}
```

:::note
STREAMINGPLUS uses S3 multipart upload for all files. The `s3:AbortMultipartUpload` permission is required to clean up incomplete uploads if the deployment restarts mid-file.
:::

## S3-Compatible Storage

The S3 sink also works with MinIO, Ceph, and other S3-compatible stores. Override the endpoint in the connection:

```bash
sp connections create minio-local \
  --type s3 \
  --endpoint http://minio.internal:9000 \
  --access-key minioadmin \
  --secret-ref minio-secret
```

Set `pathStyle: true` in the sink config if your S3-compatible store does not support virtual-hosted-style URLs.

```yaml
spec:
  sink:
    type: s3
    connectionRef: minio-local
    pathStyle: true
    bucket: my-bucket
    region: us-east-1
    format: json
```
